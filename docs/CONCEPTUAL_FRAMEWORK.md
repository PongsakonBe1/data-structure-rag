# กรอบแนวคิดเชิงแนวคิด (Conceptual Framework) และการอธิบายเทคโนโลยี

## บทที่ 1 กรอบแนวคิดเชิงแนวคิด (Conceptual Framework)

### 1.1 แนวคิดหลักของระบบ

ระบบ RAG Chatbot สำหรับเอกสารโครงสร้างข้อมูลนี้ ถูกออกแบบบนพื้นฐานของ **"Grounded-Only Multi-Modal Retrieval"** ซึ่งเป็นแนวคิดที่ผสมผสานการค้นคืนข้อมูลแบบหลายมิติ (Multi-Modal) กับนโยบายการตอบคำถามที่อิงหลักฐาน (Evidence-Based) เท่านั้น

### 1.3 องค์ประกอบหลัก 4 ประการ

#### 1.3.1 ระบบค้นคืนข้อมูลแบบไฮบริด (Hybrid Retrieval)
ผสมผสานการค้นหา 3 รูปแบบ:
- **ColPali (Visual)** - ค้นหาด้วยภาพ/แผนภาพโครงสร้างข้อมูล
- **SPLADE (Sparse)** - ค้นหาด้วยคำสำคัญและบริบท
- **BM25 (Lexical)** - ค้นหาด้วยการจับคู่คำตรงตัว

#### 1.3.2 ระบบตรวจสอบหลักฐาน (Evidence Validation)
ใช้ VLM (Vision-Language Model) ตรวจสอบว่าข้อมูลที่ค้นพบสอดคล้องกับคำถามจริงหรือไม่

#### 1.3.3 นโยบาย Grounded-Only
- ตอบได้เฉพาะเมื่อมีหลักฐานในเอกสาร
- ทุกคำตอบต้องมี citation อ้างอิง
- ปฏิเสธการตอบอย่างสุภาพเมื่อหลักฐานไม่เพียงพอ

#### 1.3.4 การประเมินคุณภาพ (Quality Assessment)
ใช้ Confusion Matrix ประเมินผล:
- TP: ถามตรง ตอบตรง
- TN: ถามนอก ไม่ตอบ (ถูกต้อง)
- FP: ถามนอก แต่ตอบ (ผิด)
- FN: ถามตรง แต่ไม่ตอบ (ผิด)

### 2.1 ColPali - Visual Document Retrieval

#### 2.1.1 หลักการทำงาน
**ColPali (Column-Paligemma)** เป็นโมเดลที่ฝึกมาสำหรับการค้นหาเอกสาร PDF โดยตรงจากภาพ (Image-to-Text Retrieval) โดยไม่ต้องแปลงเป็นข้อความก่อน

**สถาปัตยกรรม:**
```
Query Image/PDF Page
        │
        ▼
┌─────────────────────┐
│   Vision Encoder    │  ← แปลงภาพเป็น visual tokens
│   (ViT-based)       │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   Late Interaction  │  ← เปรียบเทียบ query กับทุกส่วนของภาพ
│   (MaxSim)          │
└──────────┬──────────┘
           │
           ▼
    Similarity Score
```

#### 2.1.2 ทำไมต้องใช้ ColPali?
**ปัญหาของระบบ RAG ทั่วไป:**
- เอกสาร PDF มี **ตาราง แผนภาพ โครงสร้างข้อมูล** ที่ OCR อ่านผิด
- สูตรคณิตศาสตร์ โค้ด แผนผัง ถูกแปลงเป็นข้อความแล้วเสียความหมาย

**ColPali แก้ปัญหา:**
- ค้นหาจาก **ภาพต้นฉบับ** โดยตรง
- เข้าใจ **layout และ visual context** ของเอกสาร
- เหมาะกับเอกสารโครงสร้างข้อมูลที่มี **แผนภาพประกอบ** มาก

#### 2.1.3 การใช้งานในระบบนี้
```python
# ตัวอย่างการทำงาน
from colpali_engine.models import ColPali

model = ColPali.from_pretrained("vidore/colpali-v1.2")

# 1. แปลงหน้า PDF เป็น embeddings
page_embeddings = model.encode_images(pdf_pages)

# 2. แปลงคำถามเป็น query embedding
query_embedding = model.encode_queries(["อธิบายโครงสร้าง B-Tree"])

# 3. คำนวณความคล้ายคลึง
scores = query_embedding @ page_embeddings.T
```

### 2.2 SPLADE - Sparse Neural Retrieval

#### 2.2.1 หลักการทำงาน
**SPLADE (Sparse Lexical and Expansion Model)** เป็นโมเดลที่ขยายคำสำคัญ (Term Expansion) โดยอัตโนมัติ ทำให้ค้นหาเจอเนื้อหาที่เกี่ยวข้องได้แม้ใช้คำต่างกัน

**ตัวอย่างการขยายคำ:**
```
Query: "การเรียงลำดับข้อมูล"
↓
SPLADE ขยายเป็น:
- sorting (0.85)
- algorithm (0.72)
- bubble sort (0.68)
- quick sort (0.65)
- อัลกอริทึม (0.60)
- โครงสร้างข้อมูล (0.55)
```

#### 2.2.2 ทำไมต้องใช้ SPLADE?
**ปัญหาของ BM25:**
- ค้นหาคำตรงตัวเท่านั้น
- ถ้าคำถามใช้ "การเรียงลำดับ" แต่เอกสารใช้ "sorting algorithm" จะไม่เจอ

**SPLADE แก้ปัญหา:**
- ใช้ **Neural Network** เรียนรู้ความสัมพันธ์ระหว่างคำ
- ขยายคำค้นหาโดยอัตโนมัติให้ครอบคลุมคำที่ใกล้เคียง
- เหมาะกับคำถามภาษาไทยที่มี **หลายคำศัพท์** อธิบายเรื่องเดียวกัน

#### 2.2.3 การใช้งานในระบบนี้
- ทำงานร่วมกับ ColPali แบบ **Late Fusion**
- ใช้ **RRF (Reciprocal Rank Fusion)** รวมผลคะแนนจากทั้งสองแหล่ง

### 2.3 BM25 - Lexical Matching

#### 2.3.1 หลักการทำงาน
**BM25 (Best Match 25)** เป็นอัลกอริธึมค้นหาแบบดั้งเดิมที่คำนวณความสัมพันธ์ระหว่างคำค้นหากับเอกสาร

**สูตรคำนวณ:**
```
Score(Q, D) = Σ IDF(q_i) * [f(q_i, D) * (k1 + 1)] / [f(q_i, D) + k1 * (1 - b + b * |D|/avgDL)]

Where:
- Q = Query
- D = Document
- q_i = คำที่ i ใน query
- f(q_i, D) = ความถี่ของคำ q_i ในเอกสาร D
- IDF = Inverse Document Frequency
- k1, b = parameters (ปกติ k1=1.5, b=0.75)
```

#### 2.3.2 ทำไมต้องใช้ BM25?
**บทบาทในระบบ:**
- เป็น **baseline** ที่เชื่อถือได้
- ค้นหา **คำเฉพาะทาง** เช่น "AVL Tree", "Red-Black Tree" ได้แม่นยำ
- ทำงานเร็ว ไม่ต้องใช้ GPU
- ใช้เป็น **fallback** เมื่อ Neural retrieval ไม่เจอ

#### 2.3.3 Hybrid กับ SPLADE
```
┌────────────────────────────────────────┐
│         HYBRID: BM25 + SPLADE          │
├────────────────────────────────────────┤
│ BM25: "B-Tree" → เอกสารที่มีคำนี้     │
│         ตรงตัวเท่านั้น               │
├────────────────────────────────────────┤
│ SPLADE: "ต้นไม้ทวิภาค" → ขยายเป็น    │
│         "binary tree", "B-Tree",      │
│         "tree structure"              │
├────────────────────────────────────────┤
│ RRF: รวมผล → จัดอันดับสุดท้าย         │
└────────────────────────────────────────┘
```

### 2.4 VLM (Vision-Language Model) - Evidence Grounding

#### 2.4.1 หลักการทำงาน
**VLM** เป็นโมเดลที่เข้าใจทั้ง **ภาพและข้อความ** พร้อมกัน ใช้ตรวจสอบว่าข้อมูลที่ค้นพบสอดคล้องกับคำถามจริงหรือไม่

**สถาปัตยกรรม:**
```
┌─────────────────────────────────────────────┐
│           VLM Grounding Process              │
├─────────────────────────────────────────────┤
│                                             │
│  Query: "อธิบายโครงสร้าง B-Tree"            │
│         │                                   │
│         ▼                                   │
│  ┌─────────────────┐                       │
│  │ Retrieved Page  │  ← ภาพหน้าเอกสาร      │
│  │  [หน้า 35]       │                       │
│  └────────┬────────┘                       │
│           │                                 │
│           ▼                                 │
│  ┌─────────────────────────────────────┐  │
│  │  VLM (Qwen2.5-VL / Gemini Pro Vision)│  │
│  │                                     │  │
│  │  Input:                             │  │
│  │  - ภาพหน้าเอกสาร                    │  │
│  │  - คำถาม + คำตอบที่จะให้            │  │
│  │                                     │  │
│  │  Output: Grounding Score (0-1)     │  │
│  │          Evidence Agreement          │  │
│  └─────────────────────────────────────┘  │
│           │                                 │
│           ▼                                 │
│  Decision: ผ่าน (>0.7) / ไม่ผ่าน (<0.7)     │
│                                             │
└─────────────────────────────────────────────┘
```

#### 2.4.2 ทำไมต้องใช้ VLM?
**ปัญหาของระบบ RAG ปกติ:**
- ค้นพบเอกสารที่มีคำคล้ายกัน แต่ **ไม่ตอบคำถามจริง**
- ไม่สามารถตรวจสอบว่า **แผนภาพ** ในหน้านั้นตรงกับคำถามหรือไม่

**VLM แก้ปัญหา:**
- อ่าน **แผนภาพประกอบ** ในหน้าเอกสาร
- ตรวจสอบ **ความสอดคล้อง** ระหว่างคำถามกับเนื้อหา
- ลด **Hallucination** โดยการยืนยันหลักฐานก่อนตอบ

#### 2.4.3 Consensus Voting
ใช้ **หลาย VLM** (Ensemble) ลงมติร่วมกัน:
- Qwen2.5-VL (จีน/อังกฤษ/ไทยดี)
- Gemini Pro Vision (Google)
- Claude 3 Vision (Anthropic)

```python
# Consensus voting
scores = [vlm1_score, vlm2_score, vlm3_score]
avg_score = sum(scores) / len(scores)
evidence_ok = avg_score > 0.7  # Threshold
```

## บทที่ 4 Technology Justification

### 4.1 สรุปการเลือกใช้เทคโนโลยี

| Technology | ใช้ทำอะไร | ทำไมถึงเลือก | ทดแทนอะไรได้ |
|------------|-----------|--------------|---------------|
| **ColPali** | Visual retrieval | ค้นหาจากภาพ/แผนภาพโดยตรง | CLIP, CLAP |
| **SPLADE** | Sparse neural search | ขยายคำค้นหาอัตโนมัติ | BM25 อย่างเดียว |
| **BM25** | Lexical matching | ค้นหาคำตรงตัวที่แม่นยำ | - |
| **VLM** | Evidence grounding | ตรวจสอบความถูกต้องด้วยภาพ | LLM อย่างเดียว |
| **Cross-Encoder** | Reranking | จัดอันดับละเอียด | Bi-Encoder |
| **FAISS** | Vector storage | ค้นหาเวกเตอร์เร็ว | Annoy, ScaNN |
| **LangChain** | Orchestration | จัดการ pipeline | LlamaIndex |
| **Streamlit** | UI | สร้าง interface เร็ว | Gradio, Flask |

### 4.2 การผสมผสานที่เหมาะสม (Right Combination)

**Why not ColPali alone?**
- ColPali ดีกับภาพ แต่พลาดข้อความละเอียด
- ต้องมี BM25+SPLADE เสริมเรื่องคำศัพท์เฉพาะ

**Why not BM25 alone?**
- พลาดความหมายเชิงบริบท
- ไม่เข้าใจคำที่ไม่ตรงตัว

**Why VLM after retrieval?**
- ป้องกันการตอบจากเอกสารที่ "ดูคล้าย" แต่ไม่ตรงคำถาม
- ตรวจสอบแผนภาพได้ (สำคัญมากสำหรับ CS)
